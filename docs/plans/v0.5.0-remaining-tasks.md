## T1: Feature Card 字段扩展

**目标**：FeatureCard 从简单的 title+description 升级为 5 维结构化

**改动**：

1. `src/patent.rs` — FeatureCard 加字段：
```rust
pub struct FeatureCard {
    // 原有
    pub id: String,
    pub idea_id: String,
    pub title: String,
    pub description: String,
    pub novelty_score: Option<f64>,
    pub created_at: String,
    // 新增 5 个维度
    pub technical_problem: String,    // 解决什么技术问题
    pub core_structure: String,       // 核心技术结构/方案
    pub key_relations: String,        // 关键技术关系/连接
    pub process_steps: String,        // 工艺/实施步骤
    pub application_scenarios: String, // 应用场景/领域
}
```

2. `src/db/migrations.rs` — v8→v9 迁移：
```sql
ALTER TABLE feature_cards ADD COLUMN technical_problem TEXT DEFAULT '';
ALTER TABLE feature_cards ADD COLUMN core_structure TEXT DEFAULT '';
ALTER TABLE feature_cards ADD COLUMN key_relations TEXT DEFAULT '';
ALTER TABLE feature_cards ADD COLUMN process_steps TEXT DEFAULT '';
ALTER TABLE feature_cards ADD COLUMN application_scenarios TEXT DEFAULT '';
```

3. `src/db/idea.rs` — 更新 insert/get 的字段列表

4. `src/routes/feature_cards.rs` — CreateFeatureCardRequest 加对应字段

**验收**：创建特征卡片时能传入 5 个新字段，查询时能返回

## T3: 最小差异定位增强

**目标**：api_feature_card_diff 返回差异类型分类 + 新颖性判断

**改动**：

1. `src/routes/feature_cards.rs` — diff 响应加字段：
```json
{
  "diff": { ... },
  "diff_type": "structure",       // structure | method | parameter
  "novelty_significance": true,   // 该差异是否具有新颖性意义
  "novelty_reason": "核心结构采用了不同的技术方案"
}
```

2. 差异类型判断逻辑：
   - core_structure 有差异 → "structure"
   - process_steps 有差异 → "method"
   - 其他字段有差异 → "parameter"
   - novelty_significance = 差异出现在 core_structure 或 process_steps 中

**验收**：`GET /api/feature-cards/diff?a=xx&b=yy` 返回 diff_type 和 novelty_significance
