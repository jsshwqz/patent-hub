# InnoForge v0.5.x 质量事故复盘（2026-04-18）

## 背景
用户反馈 `v0.5.3` 本地体验严重退化，且 `v0.5.1 ~ v0.5.4` 发布包异常，无法支持正常验收。

## 事实证据（已核验）
1. 标签源码并非“空仓”：
- `v0.5.0` 文件数：113
- `v0.5.1` 文件数：95
- `v0.5.2` 文件数：95
- `v0.5.3` 文件数：95
- `v0.5.4` 文件数：95

2. GitHub Release 资产异常：
- `v0.5.1` 资产数：0
- `v0.5.2` 资产数：0
- `v0.5.3` 资产数：0
- `v0.5.4` 资产数：0
- `v0.5.0` 与 `v0.5.5` 均为 5 个平台包（正常）

3. `v0.5.4` 主仓提交内容偏窄：
- 标签提交 `346e426`
- 仅改动 4 个文件：`CHANGELOG.md`、`Cargo.toml`、`README.md`、`src/lib.rs`
- 与“多终端完整可用”用户预期存在明显差距

4. 版本对照测试（源码层）：
- `v0.5.0`：`cargo test --tests` 通过
- `v0.5.3`：`cargo test --tests` 通过
- 说明：单元/集成测试通过不等于页面与端到端体验达标

## 根因
1. 发布链路把“代码通过”错误替代为“交付通过”，缺少发布资产核验门禁。
2. 发布前未执行“页面级 + 功能级 + 输入输出结果级”实测。
3. 发布后未做双端（GitHub/Gitee）资产一致性复查与失败闭环。

## 纠偏动作（强制）
1. 发布前必须执行 `docs/release-gate-manual-e2e.md` 全清单，逐项记录结果。
2. 发布后必须核对：
- GitHub Release 资产是否完整（至少 5 平台包）
- Gitee 标签/分支提交是否与 GitHub 一致
3. 任何一项失败，不得宣布发布成功，必须修复并复验。

## 当前状态
- 0.5.6 开发已暂停（按用户要求）。
- 先完成发布质量体系纠偏，再恢复功能迭代。
